## 📋 Problema & Necessidade

A plataforma NEXUS gera captions (Story 3.5) mas carece de **imagens base customizadas** para cada post. Usuários precisam de:
1. **Imagens geradas dinamicamente** (via Nando Banana API) baseadas em análise de conteúdo
2. **Processamento e branding** — adicionar logo, colors da marca, filtros
3. **Múltiplos formatos** — Instagram feed (1080×1350), Stories (1080×1920), Reels (1080×1920)

**Valor de negócio:** Content pipeline 100% automatizado (análise → caption → visual) reduz tempo production de 2hrs → 15min.

### Pontos Críticos
- **Nando Banana Rate Limit:** 100 req/min — implementar queue se excess
- **Memory usage:** Imagens 2048×2048 podem consumir 50MB cada → usar streams
- **Logo positioning:** Customizável por brand (some brands want center, other corner)
- **Color overlay:** Não pode "lavar" a imagem — test com 20-40% opacity range

### Referências
- Nando Banana API docs: https://nando-banana.dev/docs (apiKey in `.env`)
- Sharp documentation: https://sharp.pixelplumbing.com/

## 🚨 Riscos & Mitigações

| Risco | Severity | Mitigação |
|-------|----------|-----------|
| Nando Banana API offline | HIGH | Implementar fallback → imagem placeholder com colors apenas |
| Imagem processada > 30s | HIGH | Timeout + queue com async processing (user notificado) |
| Logo path inválido | MEDIUM | Validar path on startup, log error, skip logo se invalid |
| Memory leak em temp files | MEDIUM | Job scheduler limpa /tmp a cada 6hrs, manual cleanup on error |
| Color overlay distorce imagem | MEDIUM | Testes com vários backgrounds, ajustar opacity range (15-45%) |
| Cache key collision | LOW | Incluir brand_id + format + content_hash em cache key |

### Dev Notes
**Graceful Fallback Strategy:** If Nando Banana API fails after 3 retries, catch error and log. Tests ensure system doesn't crash. In production, return error to client with retry guidance.

**Cache Key Design:** SHA-256 hash of (prompt + branding config) ensures unique cache per combination. Same prompt with different colors = different cache entry.

**In-Memory vs Redis:** Currently using in-memory cache for dev/testing. Ready to swap for Redis client with same interface (get/set/delete/clearExpired).
