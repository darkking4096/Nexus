## 📐 Escopo Técnico

### AnalyticsService
```typescript
class AnalyticsService {
  async getProfileMetrics(profileId: string): Promise<Metrics>
  async getPostMetrics(profileId: string, postId: string): Promise<PostMetrics>
  async getEngagementRate(profileId: string, days: number): Promise<number>
  async scheduleMetricsCollection(): Promise<void>
}
```

### Database Schema
```sql
CREATE TABLE metrics (
  id TEXT PRIMARY KEY,
  profile_id TEXT NOT NULL REFERENCES profiles(id),
  followers_count INTEGER,
  engagement_rate DECIMAL,
  reach INTEGER,
  impressions INTEGER,
  collected_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE post_metrics (
  id TEXT PRIMARY KEY,
  content_id TEXT REFERENCES content(id),
  likes INTEGER,
  comments INTEGER,
  shares INTEGER,
  saves INTEGER,
  reach INTEGER,
  collected_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### Endpoint
**GET /api/analytics/{profileId}**
```json
Response: {

  "history": [{ "date": "2026-04-07", "followers": 14950 }],
  "comparison": { "vs_competitors": "5% below avg" }
}
```

### Key Decisions

- **Service Layer Pattern**: Reuso do padrão visto em ResearchService (DB layer encapsulado, métodos com ownership checks)
- **Metrics Collection**: Chamadas ao Python service (Instagrapi) de forma assíncrona — sem bloqueios
- **Error Handling**: "Access denied" separado de "not found" para granularidade de debug
- **Scheduler Strategy**: `collectMetrics.ts` é agnóstico a engine — pode usar node-cron, AWS Lambda, ou cron externo

### Design Pattern Applied

**Repository + Service Layer**:
```
Route (HTTP) → Service (business logic) → Model (DB) → Database
```

Ownership validation acontece em `Service`, não em `Route`. Isso permite reutilização em múltiplos contextos (HTTP, jobs, eventos).
