# Story 5.4: Best Time Optimization (Engagement-Based Scheduling)

**Epic**: 5 - Publishing Pipeline | **Status**: Draft | **Priority**: SHOULD | **Owner**: @dev | **Depends On**: 1.6

**Date Created**: 2026-04-14  
**Complexity**: Large (Timezone handling, SQL aggregation, confidence algorithm, new service)  
**Business Value**: Provide intelligent publish time recommendations based on audience engagement patterns, helping users maximize post visibility and reach.

---

## 📋 Descripción

Create an OptimizationService that analyzes historical engagement data (likes, comments, reach) by publish hour to identify the "best time" for new content. Returns recommended publish time plus confidence score.

**User Flow:**
1. Before publishing, user can request "best time" suggestion
2. System analyzes past 30 days of posts:
   - Group by hour of publish (0-23)
   - Calculate average engagement rate, reach, impressions per hour
   - Find top 3 hours with highest engagement
3. Return recommendation with timestamp and confidence
4. User can accept suggestion, pick from top 3, or ignore
5. System logs which recommendation was used (for future model training)

**Technical Context:**
- AnalyticsService (Story 1.6) already collects metrics per post
- metrics table has: `profile_id`, `collected_at`, `followers_count`, `engagement_rate`, `reach`, `impressions`
- Need to correlate with content.published_at to get publish hour
- Time-of-day analysis must be timezone-aware (user's local timezone)

### Database Queries

**Aggregate Engagement by Hour:**
```sql
SELECT
  EXTRACT(HOUR FROM c.published_at AT TIME ZONE ?) AS publish_hour,
  AVG(m.engagement_rate) AS avg_engagement,
  COUNT(DISTINCT c.id) AS post_count,
  STDDEV(m.engagement_rate) AS engagement_variance
FROM content c
JOIN metrics m ON c.id = m.content_id
WHERE c.profile_id = ?
  AND c.status = 'published'
  AND m.collected_at >= NOW() - INTERVAL '? days'
GROUP BY EXTRACT(HOUR FROM c.published_at AT TIME ZONE ?)
ORDER BY avg_engagement DESC;
```

## ⚠️ Technical Risks & Notes

**Risk 1: Insufficient Data**
- New profiles might have < 5 posts
- Mitigation: Return confidence score; user can wait for more posts
- Resolution: Clear error message when < 5 posts; suggest user publishes more

**Risk 2: Daylight Saving Time**
- DST transitions might cause hour mismatches (2 AM might not exist, 2 AM might happen twice)
- Mitigation: Use timezone-aware timestamps in database
- Resolution: Test with profile in DST-observing timezone; use database functions for time conversion

**Risk 3: Timezone Drift**
- User might change timezone; old posts use old timezone
- Mitigation: For MVP, assume static timezone; log timezone in user preferences
- Resolution: Add timezone change migration (future story)

**Risk 4: Correlation vs Causation**
- Hour with high engagement might be due to content quality, not time
- Mitigation: Confidence score reflects sample size and variance
- Resolution: Document in UI that this is a suggestion, not guarantee

**Risk 5: Global Audience**
- Audience might span multiple timezones
- Mitigation: Optimize for user's own timezone (simplest MVP)
- Resolution: Add audience timezone distribution (future story)
