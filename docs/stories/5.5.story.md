# Story 5.5: Retry & Error Handling (Exponential Backoff & Recovery)

**Epic**: 5 - Publishing Pipeline | **Status**: Draft | **Priority**: MUST | **Owner**: @dev | **Depends On**: 5.1

**Date Created**: 2026-04-14  
**Complexity**: Large (Error classification, jitter, comprehensive logging, foundational for all publish)  
**Business Value**: Provide robust error recovery for publishing failures, ensuring transient errors (network hiccups, rate limits) don't prevent successful posts while properly failing fast on permanent errors.

---

## 📋 Descripción

Implement comprehensive retry logic with exponential backoff for publish operations. This story establishes the retry framework that all publishing stories (5.1-5.3) depend on. Includes smart error classification (transient vs permanent), detailed logging for debugging, and user notifications.

**User Flow:**
1. User publishes content
2. If Playwright fails (network timeout, browser crash, etc.):
   - Automatic retry after 2 seconds
   - If fails again: retry after 4 seconds
   - If fails again: retry after 8 seconds
   - If all 3 attempts fail: stop, log error, notify user
3. User sees clear error message:
   - Transient error: "Network issue, will retry automatically"
   - Permanent error: "Instagram rejected post (duplicate caption?)" with recovery suggestion
4. Admin can view retry logs in publish_logs table for debugging

**Technical Context:**
- Retry utility already exists (Story 1.3): `utils/retry.ts` with `retryWithBackoff()`
- This story refines the error handling and expands logging
- Story 5.1 uses basic retry; 5.5 upgrades with better classification and notifications

## 🎯 Critérios de Aceitación

**Retry Mechanism:**
- [ ] Max 3 total attempts (1 initial + 2 retries)
- [ ] Exponential backoff: 2s, 4s, 8s delays between attempts
- [ ] Backoff formula: `delay = baseDelay * 2^(attemptNumber-1)` (capped at maxDelay)
- [ ] Jitter: add ±10% random variation to delay (prevent thundering herd)
- [ ] Backoff applies to all publish operations (photo, carousel, story)

**Error Classification:**
- [ ] **Transient errors** (retry automatically):
  - Network timeout (ECONNREFUSED, ETIMEDOUT)
  - HTTP 429 (rate limit)
  - HTTP 5xx (server error)
  - Playwright browser crash
  - Instagram temporarily unavailable
- [ ] **Permanent errors** (fail immediately):
  - HTTP 401/403 (auth failure, session expired)
  - Content validation failure (image too large)
  - Duplicate caption (Instagram policy)
  - Account suspended (HTTP 403 from Instagram)
  - Invalid session (decryption fails)

**Logging & Audit Trail:**
- [ ] Log every attempt: action, timestamp, result, error (if any)
- [ ] Database: publish_logs table captures all attempts
- [ ] Each attempt adds row: `{ id, profile_id, content_id, action, status, error_message, timestamp }`
- [ ] Example log sequence:
  ```
  [START] Publish content-123 (attempt 1/3)
  [ATTEMPT-1] Failed: network timeout
  [WAIT] 2 seconds + 0.2s jitter = 2.2s
  [ATTEMPT-2] Failed: HTTP 429 rate limit
  [WAIT] 4 seconds + 0.4s jitter = 4.1s
  [ATTEMPT-3] Success: post ID ig-12345, URL https://instagram.com/p/abc123/
  [PUBLISHED] content-123 -> ig-12345
  ```

**User Notifications:**
- [ ] On immediate retry: update content.status = 'retrying' (not persisted, in-memory)
- [ ] On final failure: content.status = 'error', publish_error = message
- [ ] Notification message format:
  - Transient: "⚠️ Network issue encountered. Retrying automatically (attempt 2/3)..."
  - Permanent: "❌ Failed to publish: Instagram rejected post. Reason: {specific error}"
  - Recovery suggestion: "Try again with different caption" (if duplicate), "Check session" (if auth), etc.
- [ ] WebSocket or polling endpoint: `GET /api/content/{contentId}/status` returns current status + last error

**Enhanced Implementation:**
```typescript
export async function retryWithBackoff<T>(
  fn: () => Promise<T>,
  options: RetryOptions = {}
): Promise<T> {
  const {
    maxAttempts = 3,
    baseDelayMs = 2000,
    maxDelayMs = 120000,
    jitterPercent = 10,
    isTransientError = defaultIsTransientError,
    onRetry = () => {},
  } = options;

  let lastError: unknown;

  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return await fn();
    } catch (error) {
      lastError = error;

      // Check if error is transient
      if (!isTransientError(error)) {
        throw error; // Fail fast on permanent errors
      }

      // Don't retry on last attempt
      if (attempt === maxAttempts) {
        throw error;
      }

      // Calculate backoff with jitter
      const baseDelay = Math.min(baseDelayMs * Math.pow(2, attempt - 1), maxDelayMs);
      const jitter = (Math.random() - 0.5) * 2 * (baseDelay * jitterPercent / 100);
      const delayMs = Math.max(baseDelay + jitter, 0);

      // Invoke callback
      onRetry(attempt, error);

      // Wait before retry
      await sleep(delayMs);
    }
  }

  throw lastError;
}

function defaultIsTransientError(error: unknown): boolean {
  if (error instanceof Error) {
    const msg = error.message.toLowerCase();
    // Network errors
    if (msg.includes('econnrefused') || msg.includes('etimedout') || msg.includes('network')) {
      return true;
    }
    // HTTP 429, 5xx
    if (msg.includes('429') || msg.includes('5xx') || msg.includes('500')) {
      return true;
    }
    // Playwright errors
    if (msg.includes('timeout') || msg.includes('browser closed')) {
      return true;
    }
  }
  return false;
}
```

### Error Classification

**Transient Error Detectors:**
```typescript
function isNetworkError(error: unknown): boolean {
  if (error instanceof Error) {
    return /ECONNREFUSED|ETIMEDOUT|EHOSTUNREACH|ENETUNREACH/i.test(error.message);
  }
  return false;
}

function isRateLimitError(error: unknown): boolean {
  if (error instanceof Error) {
    return /429|rate.?limit|too.?many.?requests/i.test(error.message);
  }
  return false;
}

function isServerError(error: unknown): boolean {
  if (error instanceof Error) {
    return /5\d\d|server.?error|temporarily.?unavailable/i.test(error.message);
  }
  return false;
}

// Combined checker
function isTransientError(error: unknown): boolean {
  return isNetworkError(error) || isRateLimitError(error) || isServerError(error) || isBrowserError(error);
}
```

**Permanent Error Detectors:**
```typescript
function isPermanentError(error: unknown): boolean {
  if (error instanceof Error) {
    const msg = error.message.toLowerCase();
    // Auth failures
    if (/401|403|unauthorized|forbidden|session.?expired/.test(msg)) return true;
    // Validation failures
    if (/image.?too.?large|invalid.?format|validation.?error/.test(msg)) return true;
    // Policy violations
    if (/duplicate|blocked|suspended|content.?policy/.test(msg)) return true;
  }
  return false;
}
```

### Publish Service Integration

**Update PublishService.publish() method:**
```typescript
async publish(contentId: string, profileId: string): Promise<PublishResponse> {
  const content = this.getContent(contentId);
  
  try {
    this.logAttempt(profileId, contentId, 'START', 'Publishing started');

    let publishResult: PublishResult;
    
    publishResult = await retryWithBackoff(
      async () => {
        // Determine content type and call appropriate method
        if (content.type === 'carousel') {
          return await playwrightService.publishCarousel(...);
        } else if (content.type === 'story') {
          return await playwrightService.publishStory(...);
        } else {
          return await playwrightService.publishPhoto(...);
        }
      },
      {
        maxAttempts: 3,
        baseDelayMs: 30000,  // Instagram rate limit buffer
        maxDelayMs: 120000,
        jitterPercent: 10,
        isTransientError: (err) => {
          // Custom classifier for publishing context
          return isInstagramTransientError(err);
        },
        onRetry: (attempt, error) => {
          // Log each retry attempt
          this.logAttempt(
            profileId,
            contentId,
            `RETRY-${attempt}`,
            `Attempt ${attempt + 1}/3: ${this.getErrorMessage(error)}`
          );
        }
      }
    );

    // Update database on success
    const now = new Date().toISOString();
    const stmt = this.db.prepare(`
      UPDATE content SET
        instagram_post_id = ?,
        instagram_url = ?,
        status = 'published',
        published_at = ?,
        updated_at = ?
      WHERE id = ?
    `);
    stmt.run(publishResult.postId, publishResult.url, now, now, contentId);
    
    this.logAttempt(profileId, contentId, 'PUBLISHED', `Post ID: ${publishResult.postId}`);
    
    return { ... };
  } catch (error) {
    // Error already logged by retryWithBackoff callback
    // Update content with final error
    const errorMsg = this.getErrorMessage(error);
    const stmt = this.db.prepare(`
      UPDATE content SET
        publish_error = ?,
        status = 'error',
        updated_at = ?
      WHERE id = ?
    `);
    stmt.run(errorMsg, new Date().toISOString(), contentId);
    
    this.logAttempt(profileId, contentId, 'FAILED', errorMsg);
    
    throw new Error(`Failed to publish: ${errorMsg}`);
  }
}
```

```typescript
app.get('/api/content/:contentId/status', authenticate, async (req, res) => {
  const { contentId } = req.params;
  const content = db.prepare('SELECT * FROM content WHERE id = ?').get(contentId);

    ORDER BY timestamp DESC
    LIMIT 10

  return res.json({
    content_id: contentId,
    status: content.status, // 'draft', 'publishing', 'retrying', 'error', 'published'
    error: content.publish_error,
    published_at: content.published_at,
    instagram_url: content.instagram_url,
    recent_attempts: recentLogs,
  });
});
```

## ⚠️ Technical Risks & Notes

**Risk 1: Backoff Too Conservative**
- 30s + 2s delay might be too slow for user experience
- Mitigation: Configurable delays; monitoring will show actual failure rates
- Resolution: Add metrics on retry success rate; adjust baseDelay if needed

**Risk 2: Jitter Overkill**
- ±10% jitter might not be necessary for single-user publishes
- Mitigation: Jitter still helps in multi-user scenarios; it's low cost
- Resolution: Make jitterPercent configurable if issues arise

**Risk 3: Distinguishing Error Types**
- Instagram error messages might change; regex matching might break
- Mitigation: Error classification is pluggable (isTransientError callback)
- Resolution: Monitor logs for unclassified errors; update detectors as needed

**Risk 4: Retry Logging Performance**
- Logging every retry attempt might fill database quickly
- Mitigation: publish_logs is a high-volume table; ensure proper indexing
- Resolution: Consider archiving old logs after 90 days (future story)
