# Story 3.7: Carousel Generation (SVG Overlay)

**Epic**: 3 — NEXUS Intelligence Pipeline | **Priority**: SHOULD | **Owner**: @dev | **Depends On**: 3.6 ✅

**Status**: Done | **Complexity**: M (Medium) | **Points**: 8 | **Sprint**: TBD

## 📋 Problema & Necessidade

A plataforma NEXUS pode gerar imagens para feed posts (via Nando Banana — Story 3.6), mas **não consegue criar carrosséis** (múltiplos slides com copy em overlay). Usuários precisam de carrosséis para:
- Storytelling em múltiplas cenas (narrative carousels)
- Listas com multiple items (produtos, dicas, steps)
- Comparações antes/depois

**Valor de negócio:** Carrosséis aumentam engagement em ~20% vs posts únicos. Permitir geração automática reduz tempo de edição manual.

---

## 🎯 Critérios de Aceitação

### AC-1: Geração de N Slides
```gherkin
Given um request com N imagens base (de Story 3.6 ou upload)
When POST /api/visual/generate-carousel
  { "slides": [...], "copy": [...], "brand_colors": {...} }
Then recebo HTTP 200
And response contém array de N buffers (PNG), um por slide
And cada buffer é imagem processada (1080x1350px Instagram carousel standard)
```

### AC-2: SVG Overlay com Texto
```gherkin
Given N slides com copy preparada
When aplico SVG overlay
Then texto aparece em camada SVG (não rasterizado, preserva qualidade)
And fonte responde ao brand tone (casual/profissional)
And positioning automático baseado em tamanho do texto
```

### AC-3: Contraste Garantido
```gherkin
Given imagem base com cor variável
When texto é sobreposto
Then contraste mínimo 4.5:1 (WCAG AA)
And background semi-transparente (black/white) é aplicado se necessário
And contraste é validado pix-by-pix no overlay
```

### AC-4: Numeração Automática
```gherkin
Given carousel com 5 slides
When renderizo com flag "show_numbers: true"
Then "1/5", "2/5", ..., "5/5" aparece em canto definido (bottom-right default)
And número responde a brand colors
```

### AC-5: Testes Implementados
```gherkin
Given implementação completa
Then testes cobrem:
  - [x] Geração com 1, 3, 5 slides
  - [x] Contraste validação (passing + failing cases)
  - [x] SVG overlay com diferentes tamanhos de texto
  - [x] Brand color application
  - [x] Numeração on/off
  - [x] Error handling (invalid slides, missing copy)
```

## 📚 Dev Notes

### Procedimento de Implementação
1. Usar Sharp (node image library) para processar imagens base (1080x1350px)
2. Implementar SVG overlay usando node-canvas para text rendering
3. Validar contraste pix-by-pix usando color-contrast-checker
4. Implementar numbering como feature flag (padrão: on)
5. Retornar array de Buffers (PNG) — um por slide

### Pontos Críticos
- **Contraste:** WCAG AA 4.5:1 é mandatório. Se falhar, aplicar background semi-transparente.
- **Performance:** Alvo < 2s por carousel (3 slides). Sharp é otimizado para isso.
- **Memory:** Array de Buffers pode ser grande (5 slides × 100KB ≈ 500KB). Considerar streaming em v2.
- **Font Rendering:** Node-canvas precisa de fonts do sistema. Usar fallback para sans-serif padrão.

### Referências
- Nando Banana API schema: `docs/architecture/NEXUS-schema.md`
- Brand guidelines: `docs/guides/brand-guidelines.md`
- Instagram carousel specs: `docs/guides/instagram-best-practices.md`
- Sharp docs: https://sharp.pixelplumbing.com/
- WCAG contrast: https://www.w3.org/WAI/test-evaluate/contrast-checker/

### Dependências
- `sharp` (image processing)
- `node-canvas` (SVG/text rendering)
- `color-contrast-checker` (validation)

## 🚨 Riscos & Mitigações

| Risco | Severity | Mitigação |
|-------|----------|-----------|
| Contraste falha em imagens escuras | HIGH | Aplicar white ou semi-transparent background + re-validate |
| Font rendering inconsistente | MEDIUM | Usar system fonts fallback, testar em Windows/Mac/Linux |
| Memory overflow com 10+ slides | MEDIUM | Implementar chunking, processar max 5 slides por request |
| SVG overlay rendering lento | MEDIUM | Benchmark sharp vs canvas, otimizar SVG geração |
| Numeração sobrepõe texto | LOW | Testar positioning edge cases (long text, small images) |
