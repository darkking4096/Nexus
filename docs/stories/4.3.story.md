# Story 4.3: Autopilot Configuration

**Epic**: 4 | **Priority**: MUST | **Owner**: @dev | **Depends On**: 3.10 | **Status**: Ready | **Validated**: ✅ 2026-04-13

## 📋 Descripción

Implementar configuração de Autopilot mode: permitir usuários configurarem calendário, frequência e horários de publicação automática sem necessidade de aprovação manual. Sistema valida schedule e persiste configuração por perfil.

## 🎯 Critérios de Aceitação

### Acceptance Criteria (Given/When/Then Format)

**AC 1: Criar configuração de Autopilot**
```
Given: Usuário autenticado com permissão de config
When: POST /api/profiles/{profileId}/autopilot com payload (enabled, days, times, frequency)
Then: Sistema valida payload, persiste config, retorna 200 com config_id e status "active"
```

**AC 2: Validar schedule**
```
Given: Configuração de Autopilot sendo criada/atualizada
When: Payload contém dias/horários inválidos
Then: Sistema retorna 400 Bad Request com erro específico (invalid day, invalid time format, etc)
```

**AC 3: Toggle Autopilot on/off**
```
Given: Perfil com Autopilot config existente
When: POST /api/profiles/{profileId}/autopilot/toggle?enabled=false
Then: Sistema desativa Autopilot (enabled=false), retorna 200 com nova config state
```

**AC 4: Recuperar configuração ativa**
```
Given: Perfil com Autopilot configurado
When: GET /api/profiles/{profileId}/autopilot
Then: Retorna config completa (enabled, days, times, frequency, last_updated_at, next_publication_at)
```

**AC 5: Validação de frequência**
```
Given: Configuração com dias ["MON", "WED", "FRI"] e 2 horários [09:00, 17:00]
When: Sistema calcula frequência
Then: Frequência retorna "6x per week" (3 dias * 2 horários)
```

**AC 6: Testes unitários cobrindo validações**
```
Given: Suite de unit tests para AutopilotService
When: npm test é executado
Then: Todos os testes passam (validações, toggle, CRUD operations) com 100% coverage
```

## 📐 Escopo

### IN (O que está incluído)
- AutopilotService com CRUD operations (create, read, update, delete)
- 3 endpoints: POST /api/profiles/{profileId}/autopilot, GET, POST .../toggle
- Validação de schedule (days: MON-SUN, times: HH:MM format, frequency calculation)
- Persistence em database com audit trail (created_at, updated_at)
- Toggle enable/disable sem perder config
- Unit tests com 100% code coverage
- Error handling com mensagens específicas

### OUT (O que NÃO está incluído)
- UI para configuração (delegado para Story 4.6)
- Scheduling/publication logic (delegado para Story 4.4)
- Content generation em Autopilot (delegado para Stories 3.10+)
- Analytics/reporting de publicações (future feature)

## 💼 Valor de Negócio

Permite usuários automatizar publicações em horários predefinidos sem intervenção manual, aumentando consistência e frequência de conteúdo. Feature diferencial que reduz carga operacional e melhora engagement do usuário.

## ⚠️ Riscos Documentados

| Risco | Probabilidade | Impacto | Mitigation |
|-------|--------------|--------|-----------|
| Schedule validation logic incompleto (edge cases com timezones) | MEDIUM | MEDIUM | Adicionar extensive test cases para timezone handling |
| Performance queries grandes (muitos perfis com Autopilot ativo) | LOW | MEDIUM | Indexar enabled field, criar background job para processing |
| Conflito com Manual Mode (user toggling entre modos) | MEDIUM | HIGH | Implementar state machine: only one mode active at a time |
| Database transaction issues com concurrent updates | LOW | HIGH | Usar optimistic locking, implementar test de concorrência |
