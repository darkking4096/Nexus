# Story 5.1: Feed Post Publishing (Single Photo & Carousel)

**Epic**: 5 - Publishing Pipeline | **Status**: InReview | **Priority**: MUST | **Owner**: @dev | **Depends On**: 1.3

**Date Created**: 2026-04-14  
**Complexity**: Medium (Story 1.3 already has PublishService; this extends existing logic)  
**Business Value**: Enable users to publish curated content to Instagram feed with automatic retry and error handling, supporting both single-photo and carousel formats.

---

## 📋 Descripción

Implementar a core publishing capability que suporta publicação de feed posts via Playwright automation. Suporta dois formatos: single photo (1 imagem) e carousel (múltiplas imagens).

**User Flow:**
1. User selects content (single or carousel)
2. System publishes via Playwright automation with Instagram session
3. Anti-bot delays (1-5s) between actions
4. On failure: automatic retry (max 3 attempts) with exponential backoff (2s, 4s, 8s)
5. Database updated with post ID, URL, and timestamp
6. User receives notification of success/failure

**Technical Context:**
- PublishService already exists (Story 1.3) with basic photo support
- Must extend to handle both single photos AND carousels in same `publish()` call
- PlaywrightService handles browser automation, Image hosting handles uploads
- Retry logic is in `utils/retry.ts` (exponential backoff)

### Database Interaction
**Content Table Query:**
```sql
SELECT * FROM content WHERE id = ?
-- Must return: id, profile_id, type ('photo' | 'carousel'), caption, image_url, carousel_json
```

**Update on Success:**
```sql
UPDATE content SET
  instagram_post_id = ?,
  instagram_url = ?,
  status = 'published',
  published_at = ?,
  updated_at = ?
WHERE id = ?
```

**Update on Failure:**
```sql
UPDATE content SET
  publish_error = ?,
  status = 'error',
  updated_at = ?
WHERE id = ?
```

**Publish Log:**
```sql
INSERT INTO publish_logs (id, profile_id, content_id, action, status, error_message, timestamp)
```

### Service Implementation

**Public Method:**
```typescript
async publish(contentId: string, profileId: string): Promise<PublishResponse> {
  // 1. Fetch content from database
  // 2. Validate ownership (content.profile_id === profileId)
  // 3. Get image paths (single or carousel)
  // 4. Decrypt Instagram session
  // 5. Initialize Playwright & load session
  // 6. Call retryWithBackoff() with:
  //    - publishPhoto() OR publishCarousel() 
  //    - maxAttempts: 3
  //    - baseDelayMs: 30000 (for anti-bot timing)
  // 7. On success: update content table, log attempt, return PublishResponse
  // 8. On failure: log error, update error fields, throw
  // 9. Finally: close browser (cleanup)
}
```

**Helper Methods (Existing in PublishService):**
- `getContent(contentId)` - fetch and validate
- `getDecryptedSession(profileId)` - decrypt using ENCRYPTION_KEY
- `getImagePaths(content)` - extract single or carousel images
- `logAttempt(profileId, contentId, action, status)` - audit trail
- `incrementRetryCount(contentId)` - track retries

### Retry Logic (from `utils/retry.ts`)

**Usage:**
```typescript
const result = await retryWithBackoff(
  async () => playwrightService.publishPhoto(imagePath, caption),
  {
    maxAttempts: 3,
    baseDelayMs: 30000,     // Instagram rate limit delay
    maxDelayMs: 120000      // Safety max (2 min)
  }
);
```

**Behavior:**
- Attempt 1: immediate
- Attempt 2: wait 30s, retry
- Attempt 3: wait 60s (2 × 30s), retry
- Attempt 4: wait 120s (capped), retry
- All failures: throw error

## ⚠️ Technical Risks & Notes

**Risk 1: Instagram Rate Limiting**
- Instagram may rate-limit multiple publishes in short time
- Mitigation: Use 30s base delay in backoff to simulate human behavior
- Resolution: Monitor logs for rate-limit errors; adjust backoff if needed

**Risk 2: Session Expiry**
- Instagram session might be expired when publish attempts
- Mitigation: Clear error message to user; user can refresh session via auth flow
- Resolution: Catch session error, suggest user re-authenticate

**Risk 3: Playwright Browser Crash**
- Browser might crash during publish
- Mitigation: Finally block ensures cleanup; retry logic will retry
- Resolution: Logs will show exact error; dev can investigate

**Risk 4: Image Upload Timeout**
- Large carousel with many images might timeout
- Mitigation: Test carousel with 10 images; set reasonable timeouts in PlaywrightService
- Resolution: If timeout occurs, PublishService catches and retries
