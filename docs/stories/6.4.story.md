# Story 6.4: Engagement Pattern Analysis

**Epic**: 6 | **Priority**: SHOULD | **Owner**: @dev | **Depends On**: 6.3
**Complexity**: 13 pts | **Business Value**: MEDIUM | **Status**: Ready

## 📋 Descripción

Analisar dados históricos de engagement para identificar padrões: qual horário o público engaja mais? Qual tipo de conteúdo (carousel vs reel vs post)? Qual comprimento de caption funciona? Marina usa isso para otimizar estratégia: "publicar sempre às 18h" e "fazer mais carousels".

## 🎯 Critérios de Aceitação

**Given** um perfil com 30+ posts nos últimos 60 dias  
**When** Marina requisita `GET /api/analytics/{profileId}/engagement`  
**Then**:
- [ ] Response inclui top_hours: 3 horários com maior engagement médio
- [ ] Response inclui top_content_types: ranking de tipos (carousel, reel, post, story) por engagement
- [ ] Response inclui trends: engagement está subindo/descendo ao longo do 60 dias
- [ ] Cada insight vem com % de confiança (se só 5 posts, low confidence)

**And** quando filtra por período (últimos 30 vs 60 dias)  
**Then**:
- [ ] Insights mudam baseado no período
- [ ] Trends mostram mudança de padrão entre períodos

### Detailed Algorithm Design

**Top Hours Calculation (top_hours):**
- Group posts by hour_of_day (0-23)
- Calculate weighted average engagement_rate per hour:
  - Posts in last 30 days: weight=2
  - Posts 31-60 days ago: weight=1
  - Formula: Σ(engagement × weight) / Σ(weight)
- Confidence score: min(1.0, post_count_for_hour / 30)
  - 30 posts in that hour = 1.0 confidence
  - 15 posts in that hour = 0.5 confidence
- Return top 3 hours sorted by avg_engagement

**Trends Calculation (trends):**
- Split period into two halves (0-30d and 31-60d)
- Calculate average engagement per half using weighted average
- Momentum: ((recent_avg - older_avg) / older_avg) × 100
- Direction logic:
  - If momentum > +2%: "up"
  - If momentum < -2%: "down"
  - If -2% ≤ momentum ≤ +2%: "stable"

**Confidence Scoring:**
- Base confidence on sample size: min(1.0, total_posts / 60)
- Adjust per-metric confidence by post count in subset
- Always show confidence value in response

**Data Quality Guardrails:**
- BLOCKER: Return error if profile has <30 posts
  - Response: { "error": "insufficient_data", "posts_count": 8, "min_required": 30, "recommendation": "Collect 30+ posts before requesting insights" }
- In UI, show confidence metric: "Based on X posts — low confidence" (if <30 posts in subset)
- Never display insights for hour/content_type with <3 posts

**Caption Insights (sync):**
- Keep SYNC for MVP (not background job)
- Cache results for 24h (captions don't change)
- Implement in-memory analysis: string length, emoji count correlation
- Performance: Should complete in <100ms per profile with 100 posts
- If performance budget exceeded during testing, move to background job (async endpoint)
