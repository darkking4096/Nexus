# Story 4.4: Scheduling Engine

**Epic**: 4 | **Priority**: MUST | **Owner**: @dev | **Depends On**: 4.3 | **Status**: Ready | **Validated**: ✅ 2026-04-13

## 📋 Descripción

Implementar scheduling engine para publicações: permitir agendar conteúdo para data/hora específicas com sugestão de best times baseado em histórico de engagements do perfil. Sistema persiste schedule, gerencia fila de publicações e coordena com Autopilot.

## 🎯 Critérios de Aceitação

### Acceptance Criteria (Given/When/Then Format)

**AC 1: Agendar conteúdo para data/hora específica**
```
Given: Content aprovado (ou em Autopilot mode)
When: POST /api/content/{contentId}/schedule com payload (scheduled_at: ISO8601)
Then: Sistema valida data/hora (não no passado, válido), persiste, retorna 200 com schedule_id
```

**AC 2: Validar data/hora de agendamento**
```
Given: Schedule request com data/hora inválida
When: Payload contém scheduled_at no passado OU formato inválido
Then: Sistema retorna 400 Bad Request com mensagem específica (past_date, invalid_format)
```

**AC 3: Sugerir best times baseado em histórico**
```
Given: Perfil com histórico de publicações (>5 posts)
When: GET /api/profiles/{profileId}/best-times?days=7
Then: Retorna array de 5 best times com engagement scores (highest first)
```

**AC 4: Recuperar agendamentos ativos**
```
Given: Perfil com conteúdo agendado
When: GET /api/profiles/{profileId}/scheduled
Then: Retorna lista de scheduled posts (contentId, scheduled_at, status) ordenado por scheduled_at
```

**AC 5: Atualizar agendamento**
```
Given: Scheduled content
When: POST /api/content/{contentId}/schedule com nova scheduled_at
Then: Sistema atualiza schedule, persiste mudança, retorna 200
```

**AC 6: Cancelar agendamento**
```
Given: Scheduled content
When: DELETE /api/content/{contentId}/schedule
Then: Sistema remove schedule, content volta para draft state, retorna 200
```

**AC 7: Sincronizar com fila de publicações**
```
Given: Content agendado
When: Hora de agendamento chega
Then: Sistema move de scheduled → publishing, executa publicação, atualiza queue
```

**AC 8: Testes unitários cobrendo scheduling**
```
Given: Suite de unit tests
When: npm test é executado
Then: Todos os testes passam (CRUD, validation, best-times, queue sync) com 100% coverage
```

## 📐 Escopo

### IN (O que está incluído)
- SchedulingService com CRUD operations (create, read, update, delete)
- 4 endpoints: POST/GET/DELETE /api/content/{contentId}/schedule, GET /api/profiles/{profileId}/best-times
- Validação de data/hora (não passado, formato ISO8601)
- Best times calculation baseado em últimas 7-30 dias de engagements
- Persistence em database com schedule audit trail
- Sincronização com queue (Story 4.5)
- Unit tests com 100% coverage
- Error handling específico

### OUT (O que NÃO está incluído)
- UI para scheduling (delegado para Story 4.6)
- Background job/cron para publicação automática (delegado para Story 4.4.1 ou infraestrutura)
- Timezone handling avançado (suporta UTC apenas nesta sprint)
- A/B testing de best times (future feature)

## 💼 Valor de Negócio

Permite usuários agendar publicações em horários ótimos para máximo engagements baseado em dados históricos. Reduz necessidade de estar online em horários específicos e melhora strategy de publicação com recomendações inteligentes.

## ⚠️ Riscos Documentados

| Risco | Probabilidade | Impacto | Mitigation |
|-------|--------------|--------|-----------|
| Best times calculation inaccurate (insufficient historical data) | HIGH | MEDIUM | Default fallback times, require 5+ posts before suggesting |
| Schedule synchronization fails (background job dies) | MEDIUM | HIGH | Implementar retry logic, health check monitoring |
| Concurrency issues na atualização de schedules | MEDIUM | MEDIUM | Usar database-level locking, test concurrency scenarios |
| Timezone edge cases (DST transitions) | LOW | MEDIUM | Documentar timezone limitation, suportar UTC apenas v1 |
