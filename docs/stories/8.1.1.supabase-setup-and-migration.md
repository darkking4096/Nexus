## 🧠 Dev Notes

### Current SQLite Schema Context

The existing SQLite database uses better-sqlite3 driver with the following key tables:
- **users**: user identity and authentication data
- **profiles**: user profile metadata
- **content**: generated content storage
- **analytics**: event tracking and metrics
- **cache**: temporary cache tables (non-critical)

All tables include service filtering (`.eq('service', 'ttcx')`) pattern in queries.

### Migration Strategy

1. **Phase 1: Staging Migration** — Test complete migration flow in Supabase staging environment
2. **Phase 2: Backup & Validation** — Backup production SQLite, validate schema compatibility
3. **Phase 3: Zero-Downtime Migration** — Execute migration with dual-write pattern (if needed for Story 8.1.2 compatibility)
4. **Phase 4: RLS Configuration** — Implement and test security policies
5. **Phase 5: Backup Procedure** — Document and test backup/restore workflow

### Key Technical Decisions

- **Connection String Format**: `postgresql://user:password@host/database?sslmode=require`
- **Connection Pool**: Use Supabase connection pooling (PgBouncer) to handle concurrent connections
- **RLS Pattern**: Use `current_user_id()` and `current_role()` functions for policy evaluation
- **Backup Strategy**: Daily encrypted backups with 30-day retention (Supabase managed)

### Testing Standards

- **Test Framework**: Vitest for data validation tests
- **Test Location**: `packages/backend/tests/migrations/`
- **Pattern**: Write migration validation tests that verify row counts, data integrity, and foreign key constraints
- **RLS Testing**: Test each role (service, user, admin) to ensure policies work correctly

### Important References

- Supabase Connection: https://supabase.com/docs/guides/database/connecting-to-postgres
- RLS Documentation: https://supabase.com/docs/guides/auth/row-level-security
- Migration Best Practices: https://www.postgresql.org/docs/current/sql-createtable.html
