## 📐 Escopo Técnico

### Encryption & Secrets
- Algorithm: AES-256-GCM para dados em repouso
- Password hashing: bcrypt ou Argon2 com salt >= 12 rounds
- Secrets management: GitHub Secrets / environment variables (NOT in repo)
- Key rotation: Estratégia documentada (ex: rotate JWT signing key anualmente)

### Authentication & Authorization
- OAuth 2.0 flow: Authorization Code with PKCE
- JWT tokens: HS256 ou RS256 (spec key size >= 256 bits)
- Token expiration: Access 15min, Refresh 7 days
- Refresh rotation: Novo refresh token a cada uso (invalidar anterior)
- Logout: Token blacklist ou revocation endpoint

### API Security
- Rate limiting: Implementar em middleware (express-rate-limit ou similar)
- Input validation: Sanitize todos inputs, validate schema
- SQL injection: Use parameterized queries / ORM
- XSS protection: Sanitize HTML output, use CSP headers
- CSRF protection: Implementar CSRF tokens em POST/PUT/DELETE

### Testing & Validation
- Penetration testing: Manual ou ZAP (OWASP) scan
- Unit tests: Security edge cases (expired tokens, invalid signatures, etc.)
- Integration tests: OAuth flow, token refresh, logout
- Dependency audit: `npm audit`, resolver all high/critical issues

## ⚠️ Riscos Documentados

| Risco | Impacto | Mitigação |
|-------|---------|-----------|
| Penetration testing descobre críticas | Project blocked | Começar com internal audit antes, ter plano remediation |
| Secrets leakadas em git history | Compliance issue, reputação | Usar `git-secrets` hook, rotate compromised secrets |
| JWT signing key exposure | Token forgery possível | Key rotation automática, store em KMS não filesystem |
| Rate limiting DoS falso positivo | Users legítimos bloqueados | Implement user-level limits + IP whitelisting |
| OAuth provider downtime | Users cannot login | Implement fallback auth (email+password) como backup |
| Dependency vuln não detectada | Security regression | CI gate: falha se npm audit warnings existem |
