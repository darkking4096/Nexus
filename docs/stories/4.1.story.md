# Story 4.1: Manual Mode Workflow

**Epic**: 4 | **Priority**: MUST | **Owner**: @dev | **Depends On**: 3.10 | **Status**: InProgress | **Validated**: ✅ 2026-04-13

## 📋 Descripción

Implementar workflow de Manual Mode: pesquisa → análise → caption → visual → publicação, com aprovação em cada step. Usuários podem revisar, editar e aprovar cada etapa do pipeline de conteúdo antes da publicação.

## 🎯 Critérios de Aceitação

### Acceptance Criteria (Given/When/Then Format)

**AC 1: Criar conteúdo em modo Manual**
```
Given: Um usuário autenticado com permissão de criação de conteúdo
When: Envia POST /api/content/generate com mode=manual
Then: Sistema retorna contentId com status "research_pending" e approval_panel com step 1
```

**AC 2: Aprovar/Rejeitar/Editar cada step**
```
Given: Content em step de aprovação (research, analysis, caption, visual, publish)
When: Usuário submete approval decision (approve/reject/edit) para o contentId
Then: Sistema avança para próximo step OU retorna em estado anterior com comentários
```

**AC 3: Salvar drafts**
```
Given: Conteúdo em qualquer step do workflow
When: Usuário salva draft sem enviar para aprovação
Then: Sistema persiste estado atual sem avançar workflow e retorna save confirmation
```

**AC 4: Endpoint retorna approval panel estruturado**
```
Given: Content em manual mode
When: GET /api/content/{contentId}/approval-panel
Then: Retorna estrutura completa com current_step, all_steps_history, editable_fields, action_buttons
```

**AC 5: Testes E2E cobrindo workflow completo**
```
Given: Suite de testes E2E
When: Executa fluxo research → analysis → caption → visual → publish com aprovações
Then: Todos os testes passam (100% success rate) e logs não contêm warnings/errors
```

## 📝 Dev Notes

### Architecture Decisions

**1. Storage Strategy (Opção B: workflow_states table)**
- Separate table with 1-to-1 relationship to content table
- Rationale: Clear separation of concerns, better auditability with workflow_history table, easier to query pending approvals
- Schema: `workflow_states` (schedule_data, content_data, publish_data as JSON), `workflow_history` (audit trail)

**2. State Machine (9 States)**
```
schedule_pending → schedule_approved → content_pending → content_approved 
→ publish_pending → published
With rejection → rejected (restarts from schedule_pending)
And drafts keep step unchanged
```
- Rationale: Explicit states make transitions clear, audit trail records who approved what

**3. Approval Flow (3-Stage Validation)**
1. **Schedule**: When to publish (days, times, content types)
2. **Content**: Caption + visuals ready
3. **Publish**: Final confirmation before posting

**4. Auto-Approval**
- Enabled by default (24-hour timeout)
- Rationale: User can disable or customize timeout; if no action → automatically approved
- Implementation: `checkAutoApprovals()` checks pending items with expired timeouts

**5. Rejection Behavior**
- Rejects all downstream steps, restarts from schedule_pending
- Logs reason in workflow_history for audit trail

### Implementation Details

- **ManualWorkflow Service**: 400+ lines, 5 core methods
  - `initializeWorkflow()`: Start 3-stage process
  - `submitApproval()`: Approve/reject/edit with validation
  - `saveDraft()`: Persist without advancing
  - `getApprovalPanel()`: Frontend data structure
  - `checkAutoApprovals()`: Async auto-approval processor
