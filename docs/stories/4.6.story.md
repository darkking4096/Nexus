# Story 4.6: Mode Toggle & UX

**Epic**: 4 | **Priority**: MUST | **Owner**: @ux-design-expert | **Depends On**: 4.1, 4.3 | **Status**: Ready | **Validated**: ✅ 2026-04-13

## 📋 Descripción

Implementar UI para toggle entre Manual e Autopilot modes com indicadores visuais claros de modo ativo, panels de configuração específicos para cada modo, e controls intuitivos. Interface responsive com controles para ligar/desligar cada modo e acessar configurações.

## 🎯 Critérios de Aceitação

### Acceptance Criteria (Given/When/Then Format)

**AC 1: Renderizar toggle Manual ↔ Autopilot**
```
Given: ModeToggle component
When: Component renderiza
Then: Exibe 2 buttons/tabs (Manual, Autopilot), um modo sempre ativo (highlighted)
```

**AC 2: Alternar entre modos**
```
Given: Manual mode ativo
When: Usuário clica no button Autopilot
Then: Sistema transiciona para Autopilot, Manual é deativado, UI atualiza para Autopilot panel
```

**AC 3: Manual Mode panel mostra approval workflow**
```
Given: Manual mode ativo
When: ManualModePanel é renderizado
Then: Exibe steps de workflow (research, analysis, caption, visual, publish), current step, action buttons
```

**AC 4: Autopilot panel mostra configuração de schedule**
```
Given: Autopilot mode ativo
When: AutopilotPanel é renderizado
Then: Exibe form com schedule config (days, times, frequency), toggle enable/disable, next publication time
```

**AC 5: Ligar/desligar Autopilot sem sair do mode**
```
Given: Autopilot mode ativo com config definida
When: Usuário clica toggle enable/disable
Then: Sistema persiste change, Autopilot ativa/desativa sem sair do mode, feedback visual imediato
```

**AC 6: Indicador visual de modo ativo**
```
Given: ModeToggle renderizado
When: Em qualquer modo
Then: Modo ativo tem indicador visual distinto (color, badge, underline) e label claro
```

**AC 7: Responsivo em todos os breakpoints**
```
Given: Suite de testes responsivos (mobile, tablet, desktop)
When: Cada breakpoint testado
Then: Toggle e panels adaptam layout, sem scroll horizontal, buttons acessíveis
```

**AC 8: Testes de componentes**
```
Given: Suite de tests para ModeToggle, ManualModePanel, AutopilotPanel
When: npm test é executado
Then: Todos os testes passam (mode switching, rendering, integration) com 90%+ coverage
```

## 📐 Escopo

### IN (O que está incluído)
- 3 componentes React: ModeToggle, ManualModePanel, AutopilotPanel
- Mode switching logic (Manual ↔ Autopilot, only one active)
- Visual indicators para modo ativo (color, badge, styling)
- Integração com Manual workflow (Story 4.1 approval panel)
- Integração com Autopilot config (Story 4.3 schedule form)
- Responsive CSS/Tailwind para 3 breakpoints
- Unit + integration tests com 90%+ coverage
- Accessible controls (ARIA labels, keyboard navigation)

### OUT (O que NÃO está incluído)
- Backend mode state management (apenas frontend state)
- Approval panel logic (delegado para Story 4.2)
- Schedule configuration form (delegado para Story 4.3)
- Queue management UI (delegado para Story 4.5)
- Analytics/insights panel (future feature)

## 💼 Valor de Negócio

Interface central que permite usuários trocar facilmente entre modo manual (controle total) e Autopilot (automação), adaptando fluxo de trabalho conforme necessário. Reduz confusão sobre qual modo está ativo e fornece acesso rápido a configurações de cada modo.

## ⚠️ Riscos Documentados

| Risco | Probabilidade | Impacto | Mitigation |
|-------|--------------|--------|-----------|
| Mode state mismatch (frontend vs backend) | MEDIUM | HIGH | Implementar server sync na mount, state reconciliation |
| Usability confusion se modo ativo não é claro | MEDIUM | MEDIUM | Adicionar redundant visual indicators (color + badge + label) |
| Integration issues com approval panel (Story 4.2) | MEDIUM | MEDIUM | Usar mock data, test com real approval panel early |
| Performance issues se panels fazem heavy computations | LOW | LOW | Keep panels lazy-loaded, defer expensive calculations |
