# Story 4.5: Content Queue Management

**Epic**: 4 | **Priority**: SHOULD | **Owner**: @dev | **Depends On**: 4.4 | **Status**: Ready | **Validated**: ✅ 2026-04-13

## 📋 Descripción

Implementar gerenciamento de fila de publicações: visualizar todos os posts agendados, reordenar via drag-drop, deletar de fila, e ver status em tempo real. Sincroniza com SchedulingService e exibe fila unificada de pending e scheduled posts.

## 🎯 Critérios de Aceitação

### Acceptance Criteria (Given/When/Then Format)

**AC 1: Visualizar fila de conteúdo**
```
Given: Perfil com posts agendados e pending
When: GET /api/profiles/{profileId}/queue
Then: Retorna lista unificada com scheduled (próximos 30 dias) e pending (não-agendados)
```

**AC 2: Reordenar fila (drag-drop)**
```
Given: Frontend renderiza queue com múltiplos posts
When: Usuário arrasta post e soltar em nova posição
Then: Sistema reordena, persiste nova sequência, retorna 200 com nova queue order
```

**AC 3: Deletar post da fila**
```
Given: Post em status "scheduled" ou "pending"
When: DELETE /api/profiles/{profileId}/queue/{contentId}
Then: Sistema remove de fila, persiste mudança, post volta para draft, retorna 200
```

**AC 4: Filtrar fila por status**
```
Given: Fila com posts em múltiplos status (scheduled, pending, publishing, published)
When: GET /api/profiles/{profileId}/queue?status=scheduled
Then: Retorna apenas posts com status solicitado
```

**AC 5: Paginação da fila**
```
Given: Perfil com >50 posts na fila
When: GET /api/profiles/{profileId}/queue?page=1&limit=20
Then: Retorna 20 posts com metadata (total_count, has_more, current_page)
```

**AC 6: Real-time queue updates (WebSocket)**
```
Given: Cliente conectado em WebSocket /queue/{profileId}
When: Outro cliente modifica queue (reorder, delete, add)
Then: Cliente recebe update event com nova queue state
```

**AC 7: Testes para queue operations**
```
Given: Suite de unit tests
When: npm test é executado
Then: Todos os testes passam (CRUD, reorder, pagination, sync) com 90%+ coverage
```

## 📐 Escopo

### IN (O que está incluído)
- QueueService com read, reorder, delete operations
- 3 endpoints: GET /api/profiles/{profileId}/queue, POST .../queue/reorder, DELETE .../queue/{contentId}
- Unificação de pending + scheduled posts
- Reorder persistence com audit trail
- Filtering por status e paginação
- WebSocket support para real-time updates (opcional v1)
- Unit + integration tests com 90%+ coverage
- Frontend ContentQueue component com drag-drop UI (delegado para Story 4.2 UX)

### OUT (O que NÃO está incluído)
- UI/UX (delegado para Story 4.2)
- Advanced sorting (sort by date, engagement, etc.)
- Bulk operations (delete multiple, reorder multiple)
- Archive/history de publicações (future feature)
- Performance analytics na fila (future feature)

## 💼 Valor de Negócio

Fornece visibilidade completa sobre pipeline de conteúdo e permite usuários gerenciar publicações de forma flexível. Reordenar fila habilita usuários priorizar posts baseado em eventos, tendências ou mudanças de estratégia sem cancelar agendamentos.

## ⚠️ Riscos Documentados

| Risco | Probabilidade | Impacto | Mitigation |
|-------|--------------|--------|-----------|
| Reorder concurrency (2 users reordering simultaneously) | MEDIUM | HIGH | Implementar optimistic locking com version field |
| Queue state inconsistency (delete durante reorder) | MEDIUM | HIGH | Usar transactions, implementar test de concurrency |
| Performance degradation (muitos posts em fila) | LOW | MEDIUM | Implementar paginação, add database index |
| WebSocket memory leak se clients desconectam abruptamente | MEDIUM | MEDIUM | Implementar connection timeout + cleanup, test com 100+ clients |
